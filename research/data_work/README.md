# Работа с данными

В данном разделе представлены файлы, связанные с подготовкой корпуса правил настольных игр: файлы мета-данных, скрипты парсинга правил и извлечения/очистки текстов, а также EDA-ноутбук для контроля качества данных перед дальнейшей индексацией.

## Структура

- `metadata/`
  - `docs.csv` - основной каталог документов (используется как метаданные в пайплайне подготовки и в EDA).
  - `rules_index.csv` - индекс правил/ссылок (каталог правил игр).
  - `game_docs.csv` - связь игр с документами.
  - `rules_texts_cleaned_report.csv` - отчёт по очистке (генерируется скриптом `rules_cleaner.py`).
- `scripts/`
  - `rules_parser.py`, `rules_text_extractor.py` — скрипты для получения/извлечения правил.
  - `rules_cleaner.py` - очистка извлечённых текстов правил и формирование отчёта.
- `artifacts/` - технические артефакты парсинга (например, списки успешных/неуспешных ссылок).
- `EDA.ipynb` - исследовательский ноутбук.

> Данные после выполнения скриптов появятся в корне репозитория в директории `data/`.

## Пайплайн очистки (`rules_cleaner.py`)

Скрипт читает `.txt` из `data/rules_texts/` и раскладывает результаты в:

- `data/rules_texts_cleaned_good/` — «хорошие» очищенные тексты
- `data/rules_texts_cleaned_quarantine/` — тексты, помеченные как «карантин» по quality-gate

Очистка включает:

- Нормализацию переводов строк и Unicode (NFKC), удаление невидимых/управляющих символов
- Удаление строк-маркеров страниц и «мусора», схлопывание пробелов
- Dehyphenation (склейка переносов по дефису)
- Механизм `global boilerplate`: поиск повторяющихся строк в начале/конце документов (по языкам) и удаление их при очистке
- **Границы абзацев перед заголовками**: вставка двойного переноса (`\n\n`) перед строками-заголовками разделов (например «ФАЗА N», «Подготовка к игре», «Компоненты», «Игровой процесс» и т.п.), чтобы при последующем чанкинге заголовок оказывался в начале своего абзаца/чанка, а не в конце предыдущего
- Фильтрацию абзацев по эвристикам качества (доля букв, повторяемость символов, энтропия); **абзацы, совпадающие с заголовком раздела, сохраняются** и не считаются «плохими», даже если короткие

Документ помечается как `quarantine`, если очищенный текст пустой, либо `survival_ratio` ниже порога, либо доля «плохих» абзацев превышает порог.

## Идентификаторы и хэши

В метаданных используются SHA-256 хэши как стабильные идентификаторы документов/текстов.

- `doc_sha256` в `metadata/docs.csv` и `metadata/game_docs.csv` — базовый идентификатор документа (используется как имя файла для сырого извлечённого текста: `data/rules_texts/{doc_sha256}.txt`).
- `raw_doc_sha256` в `metadata/rules_texts_cleaned_report.csv` — тот же идентификатор исходного (raw) документа/текста; cleaned-файлы сохраняются под тем же именем для удобной трассировки: `data/rules_texts_cleaned_good/{raw_doc_sha256}.txt`.
- `clean_sha256` — хэш содержимого очищенного текста.
